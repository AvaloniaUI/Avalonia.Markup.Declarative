# Avalonia.Markup.Declarative Release Notes

## Avalonia 12 refactor

This release is the largest conceptual cleanup in the project so far. The work done across the `avalonia-12` branch moves Avalonia.Markup.Declarative away from the old MVU-specific runtime, reflection-driven bindings, and custom change-handler flow. The library is now intentionally much closer to Avalonia itself: `ViewBase`, `DataContext`, compiled bindings, regular styles and selectors, and build-time source generation.

The result is a thinner public API, better type-safety, less runtime magic, and a much smaller maintenance surface for framework and third-party extension generation.

## What changed

- The MVU-oriented `ComponentBase` runtime was removed.
- Generated property and style setters now use compiled bindings based on `Expression<Func<TViewModel, TValue>>`.
- Binding authoring now follows Avalonia concepts more closely: explicit binding sources, `DataContext`, `BindingMode`, `IValueConverter`, `IBinding`, and typed styles.
- Third-party control extensions are generated by the built-in source generator instead of a separate CLI tool.
- View composition now uses `ViewFactory.Create<T>()` instead of the old component factory helper.
- Runtime diagnostics were improved so build failures can include file, line, and target-property context.

## Why this matters

Older versions of the library mixed two different mental models:

- an MVU-style component runtime with reflection-based state tracking, `StateHasChanged()`, and custom binding callbacks
- an Avalonia-style view model and binding world based on `DataContext`, bindings, styles, and selectors

This refactor chooses the second model as the foundation. The library still lets you describe UI in C#, but it now does that by staying much closer to native Avalonia behavior.

## Breaking changes

- `ComponentBase`, `ComponentBase<TViewModel>`, `IMvuComponent`, `FuncComponent`, `FuncView`, `InjectAttribute`, and the old property-state helpers were removed. If you relied on the MVU runtime, move to `ViewBase` or `ViewBase<TViewModel>`.
- `StateHasChanged()`, `UpdateState(...)`, `AddExternalState(...)`, `Bind(...)`, and `MvuBinding` are no longer part of the supported programming model. Use compiled bindings, normal Avalonia binding modes, and regular events or commands.
- Generated expression-binding setters no longer use `Func<T>` plus `Action<T>? onChanged`. They now expose compiled-binding overloads that accept `Expression<Func<TViewModel, TValue>>` and optional `BindingMode` and `IValueConverter` parameters.
- `ComponentBase.New<T>()` was replaced by `ViewFactory.Create<T>()`. If your controls need DI-aware construction, configure `UseComponentControlFactory(...)` on `AppBuilder`.
- The standalone `AvaloniaExtensionGenerator` workflow is superseded by the integrated source generator. Do not keep CLI generation steps in your build or README.
- Checked-in generated extension files are no longer the source of truth for supported assemblies. Extensions are generated during build from the referenced Avalonia packages and from opted-in external assemblies.
- `ViewBase<TViewModel>` now follows a clearer initialization model. Parameterless views initialize lazily after a compatible `DataContext` is assigned or after the view is attached. If you relied on constructor-time build, provide the view model in the base constructor or configure `ViewInitializationStrategy.Immediate`.

## Recommended patterns now

- Single-file components: `ViewBase<TState>` with a nested `ObservableObject` state class.
- MVVM views: `ViewBase<TViewModel>` with compiled bindings such as `.Text(vm, x => x.Message)` or `.Text<TViewModel>(x => x.Message)`.
- Two-way input: compiled bindings with explicit `BindingMode.TwoWay` instead of custom callbacks.
- Styles: standard Avalonia selectors plus compiled style setters such as `.IsEnabled(default(TabVm)!, x => x.Enabled)`.
- View composition: `ViewFactory.Create<T>()`.
- External libraries: `[assembly: GenerateMarkupExtensionsForAssembly(typeof(SomeTypeFromExternalAssembly))]`.

## Before -> after

### 1. MVU component to stateful declarative view

Before:

```csharp
public class CounterComponent : ComponentBase
{
    public decimal? Counter { get; set; }

    protected override object Build() =>
        new StackPanel()
            .Children(
                new TextBlock()
                    .Text(() => $"Counter: {Counter}"),
                new Button()
                    .Content("Increment")
                    .OnClick(_ =>
                    {
                        Counter++;
                        StateHasChanged();
                    })
            );
}
```

After:

```csharp
using CommunityToolkit.Mvvm.ComponentModel;

public partial class CounterComponent() : ViewBase<CounterComponent.State>(new State())
{
    protected override object Build(State state) =>
        new StackPanel()
            .Children(
                new TextBlock()
                    .Text(state, x => x.CounterLabel),
                new Button()
                    .Content("Increment")
                    .OnClick(_ => state.Counter++)
            );

    public sealed partial class State : ObservableObject
    {
        [ObservableProperty]
        [NotifyPropertyChangedFor(nameof(CounterLabel))]
        public partial decimal? Counter { get; set; } = 0;

        public string CounterLabel => Counter is null || Counter == 0
            ? "Counter: zero"
            : $"Counter: {Counter}";
    }
}
```

### 2. Custom `onChanged` callback to compiled two-way binding

Before:

```csharp
new NumericUpDown()
    .Value(() => Counter, value => Counter = value);
```

After:

```csharp
new NumericUpDown()
    .Value(state, x => x.Counter, BindingMode.TwoWay);
```

The old model pushed user edits through custom callbacks. The new model uses normal Avalonia two-way binding semantics.

### 3. Reflection-based `Bind(...)` to explicit compiled binding

Before:

```csharp
new TextBox()
    .Text(Bind(Message, BindingMode.TwoWay));
```

After:

```csharp
new TextBox()
    .Text(state, x => x.Message, BindingMode.TwoWay);
```

This removes caller-expression parsing and reflection from the main binding path.

### 4. String/path style binding to compiled style binding

Before:

```csharp
new Style<TabItem>()
    .IsEnabled(new Binding(nameof(TabVm.Enabled)))
    .Foreground(Brushes.YellowGreen);
```

After:

```csharp
new Style<TabItem>()
    .IsEnabled(default(TabVm)!, x => x.Enabled)
    .Foreground(Brushes.YellowGreen);
```

Typed style setters now follow the same compiled-binding approach as regular controls.

### 5. `ComponentBase.New<T>()` to `ViewFactory.Create<T>()`

Before:

```csharp
protected override object Build() =>
    ComponentBase.New<CounterComponent>();
```

After:

```csharp
protected override object Build() =>
    ViewFactory.Create<CounterComponent>();
```

If the control requires DI-aware construction, register a component control factory once on `AppBuilder`.

### 6. External CLI generator to integrated source generator

Before:

```bash
dotnet tool install --global AvaloniaExtensionGenerator
cd c:\your\project
avalonia-amd-gen
```

After:

```csharp
using Avalonia.Markup.Declarative;
using ReactiveUI.Avalonia;

[assembly: GenerateMarkupExtensionsForAssembly(typeof(RoutedViewHost))]
```

The build now generates external assembly extensions together with the rest of the project. No extra CLI step is required.

## Migration checklist

- Replace `ComponentBase` with `ViewBase` or `ViewBase<TViewModel>`.
- Move local mutable state into a view model or a nested `ObservableObject` state class.
- Replace `Func<T>` plus `onChanged` setter calls with compiled bindings and `BindingMode.TwoWay` where needed.
- Replace `Bind(...)` calls with generated compiled-binding overloads.
- Replace `ComponentBase.New<T>()` with `ViewFactory.Create<T>()`.
- If your views use constructor injection, configure `UseComponentControlFactory(...)` and optionally `UseServiceProvider(...)`.
- Remove `AvaloniaExtensionGenerator` tool installation and execution from your docs and build scripts.
- Add `GenerateMarkupExtensionsForAssembly` attributes for third-party assemblies that should get markup extensions.

## Notes for maintainers

This refactor was driven by the branch history that moved the project to Avalonia 12, removed obsolete extensions and the standalone generator, introduced source-less compiled bindings, refactored style and property generators, added `ViewFactory` registration support, and improved diagnostics around control creation and build errors.
