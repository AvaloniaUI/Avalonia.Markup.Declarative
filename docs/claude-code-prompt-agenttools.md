# Бриф для Claude Code: пакет `Declarative.Avalonia.AgentTools`

> Вставь этот файл целиком как стартовый промпт в Claude Code, открыв его в корне репозитория `Avalonia.Markup.Declarative`.

## Цель

Добавить в монорепозиторий инструмент обратной связи для ИИ-агентов, разрабатывающих UI на этой библиотеке: in-process **MCP-сервер**, который в режиме отладки отдаёт агенту скриншот окна или отдельного контрола, текстовое визуальное дерево с границами и именами, а также ошибки сборки и биндингов. Сервер живёт внутри запущенного отладочного приложения (под `dotnet watch`) и общается с агентом по loopback.

Перед написанием кода **сначала изучи существующий репозиторий** — мы переиспользуем то, что уже есть, и не дублируем его.

## Что прочитать в первую очередь

- `src/Avalonia.Markup.Declarative/ViewBase.cs` — стратегии инициализации (`ViewInitializationStrategy`), ленивый `Child`/`EnsureInitialized`, NameScope (`Scope`), регистрация инстанса в `OnAttachedToVisualTree`.
- `src/Avalonia.Markup.Declarative/HotReloadManager.cs` — уже ведёт реестр живых прикреплённых `ViewBase`-инстансов. Это источник для `list_components`.
- `src/Avalonia.Markup.Declarative/ControlPropertyExtensions.cs` — `_set` с `[CallerFilePath]`/`[CallerLineNumber]`, `ViewBuildingException` (уже несёт control/file/line), `Name(...)`, `Ref(out)`.
- `src/Avalonia.Markup.Declarative/DeclarativeBindingAutoConverter.cs` — кидает `InvalidOperationException` при провале конверсии; это один из источников ошибок для буфера.
- Сверься с фактической версией Avalonia в репозитории — выбор API для перехвата ошибок биндингов зависит от неё (см. ниже).

## Архитектурное решение (соблюдать строго)

Монорепо, **два пакета**. Граница проходит по зависимостям, а не по удобству:

1. **Core (`Avalonia.Markup.Declarative`)** получает только тонкую diagnostics-поверхность **без единой новой зависимости** (всё опирается на уже подключённую Avalonia). Сюда идёт логика, которой нужна лишь Avalonia: обход визуального дерева, захват контрола в `RenderTargetBitmap`, кольцевой буфер ошибок, перехват ошибок биндингов, маршалинг на UI-поток. Эта поверхность самоценна: её переиспользуют тесты и будущий headless-харнес.

2. **Новый пакет `Declarative.Avalonia.AgentTools`** содержит MCP-обвязку: транспорт, регистрацию тулов, метод `.UseAgentInspector()` и зависимость на MCP SDK. **Категорически не клади это в core.** NuGet-зависимости не условны по конфигурации — даже под `#if DEBUG` зависимость пакета протекает всем потребителям в Release, тащит MCP/web-стек, ломает trimming/AOT и даёт плохой сигнал security-ревью. AgentTools потребитель подключает явно и только в dev.

Имя пакета — именно `Declarative.Avalonia.AgentTools` (тот же неймспейс, что у существующего `Declarative.Avalonia.Templates`), чтобы не упираться в зарезервированный префикс `Avalonia.*` организации `avaloniaui`.

**Не делать:** виртуального DOM/reconciler (библиотека строит реальные контролы синхронно); биндинга сервера на не-loopback адреса; ломающих изменений публичного API core.

## Deliverable 1 — diagnostics-поверхность в core (deps-free)

Создай, например, неймспейс `Avalonia.Markup.Declarative.Diagnostics` со следующими возможностями. Реализация должна работать на UI-потоке (вызывающая сторона отвечает за маршалинг; см. Deliverable 2).

- **Обход дерева.** `string DumpVisualTree(Visual root, options)` — отступами по глубине, на каждый узел: тип, `Name` (если есть), `Bounds` (x, y, w, h), несколько ключевых свойств (например, для `TextBlock` — `Text`, для контейнеров — фон/ориентация). Границы критичны: по ним агент диагностирует layout без пикселей (нулевой размер, off-screen, перекрытие).
- **Захват контрола.** `byte[] CaptureControlPng(Control control)` через `RenderTargetBitmap.Render(control)` → PNG. Учти: рендер перерисовывает визуал, поэтому работает даже для свёрнутого/перекрытого окна. Поддержи два режима: контрол в изоляции (его сабтри) и «как в контексте» (рендер окна + кроп по `control.Bounds`).
- **Захват окна.** `byte[] CaptureTopLevelPng(TopLevel top)`.
- **Резолв по имени.** Найти контрол по `Name` в пределах конкретного view через его NameScope (`Scope`) или обходом дерева.
- **Реестр компонентов.** Тонкая обёртка над реестром живых инстансов из `HotReloadManager` → перечисление активных view (тип, имя, корневой контрол).
- **Буфер ошибок.** Потокобезопасный кольцевой буфер. Источники: перехваченные `ViewBuildingException` (control/property/file/line — это уже готовый, ценный для агента текст), исключения `DeclarativeBindingAutoConverter`, и ошибки биндингов Avalonia. Для биндингов используй механизм, который поддерживает текущая версия Avalonia в репо: предпочтительно `BindingDiagnostics` (Avalonia 11.1+), иначе кастомный `ILogSink` на `LogArea.Binding`. Каждая запись: timestamp, severity, source (control/path), сообщение.

Эта поверхность не должна тянуть ничего, кроме Avalonia. `RenderTargetBitmap` и PNG-энкодинг — часть Avalonia; рендер-бэкенд присутствует в реальном приложении в рантайме (для headless-тестов понадобится `Avalonia.Headless.Skia` — это в тестовом проекте, не в core).

## Deliverable 2 — пакет `Declarative.Avalonia.AgentTools` (in-process MCP, только debug)

Новый проект `src/Declarative.Avalonia.AgentTools/Declarative.Avalonia.AgentTools.csproj`, добавить в solution. Зависит от core-пакета и от официального C# MCP SDK (`ModelContextProtocol`).

### Точка входа

Метод-расширение `AppBuilder UseAgentInspector(this AppBuilder builder, Action<AgentInspectorOptions>? configure = null)`:
- запускает MCP-сервер на фоновом потоке при старте приложения;
- `AgentInspectorOptions`: `Port` (по умолчанию loopback, напр. 5599), `EnableInteraction` (по умолчанию `false` — см. тир-2), и т.п.;
- биндинг **только на 127.0.0.1**;
- при старте печатает в debug-консоль URL/порт, чтобы разработчик прописал его в MCP-клиент агента.

В образце приложения вызов оборачивай в `#if DEBUG`. Сам метод — явный opt-in; это и есть гейт. В Release пакет просто не подключается.

### Транспорт

HTTP/SSE (streamable) на loopback — практичный вариант для подключения внешнего агента к живому GUI-процессу. Сделай транспорт/порт настраиваемыми. Картинки возвращай **как image-content блоки (base64 PNG)**, чтобы мультимодальный агент их видел, плюс короткий text-блок с метаданными (имя контрола, размер, bounds).

### Потоки (центральный момент)

Сервер крутится на фоновом потоке, а обход дерева и `RenderTargetBitmap` обязаны идти на UI-потоке. Каждый обработчик тула маршалит работу через `Dispatcher.UIThread.InvokeAsync(...)`, ждёт результат и сериализует обратно. В документации честно отметь: если разработчик стоит на брейкпоинте, UI-поток не прокачивается и капча повиснет до возобновления — для автономного агентского цикла это нормально.

### Набор тулов

- `get_visual_tree(rootName?)` → текстовое дерево с именами и bounds (через core).
- `list_components()` → активные view из реестра.
- `screenshot_window(windowId?)` → PNG активного окна/TopLevel.
- `screenshot_control(name, mode?)` → PNG конкретного контрола (`mode`: isolated | in_context).
- `get_errors(sinceTimestamp?)` → записи из буфера ошибок (build + binding + converter).
- **Тир-2, за отдельным флагом `EnableInteraction`:** `invoke(name, action)` — клик/установка значения/фокус. Это remote-control-поверхность, поэтому выключена по умолчанию и включается только явно. Помести её за дополнительным гейтом и отметь в доке риски.

Тулы композируются: `get_visual_tree` отдаёт имена → агент выбирает → `screenshot_control(name)`. Идентификация по `Name` опирается на существующий `.Name(...)`; в доке посоветуй агенту именовать контролы, которые он итерирует.

## Безопасность и гейтинг (чек-лист)

- AgentTools подключается только в dev; в Release-граф зависимостей не попадает.
- Сервер слушает строго loopback.
- `invoke` выключен по умолчанию, включается отдельным флагом.
- Скриншоты окна потенциально содержат данные — это ещё одна причина loopback + debug-only.

## Тесты

Тестовый проект на `Avalonia.Headless` + `Avalonia.Headless.Skia`:
- отрендерить эталонный view, проверить, что `DumpVisualTree` содержит ожидаемые имена и непустые bounds;
- `CaptureControlPng`/`CaptureTopLevelPng` дают непустой валидный PNG;
- намеренно бросить `ViewBuildingException` и сделать заведомо битый биндинг — убедиться, что буфер ошибок их поймал;
- резолв контрола по `Name` находит нужный узел.

## Упаковка (подготовить метаданные, НЕ публиковать)

В `.csproj` пакета AgentTools:
- зависимость на `Avalonia.Markup.Declarative` (так пакеты кластеризуются через граф зависимостей и видна совместимость);
- одинаковые с core `RepositoryUrl`/`PackageProjectUrl`, осмысленный `Description` с явной отсылкой к `Avalonia.Markup.Declarative`, релевантные `PackageTags`;
- лицензия через `<PackageLicenseExpression>`/`license`-элемент, не `licenseUrl`;
- версионирование: выровняй мажор AgentTools с версией core, которую он инспектирует (или задай вменяемый диапазон зависимости), чтобы «какой AgentTools к какому Markup.Declarative» не было загадкой.

Публикацию и резервирование префикса не делай — это ручные шаги владельца.

## Опциональные follow-ups (только если останется время; делать отдельными PR)

1. **Headless preview-харнес** для детерминированных/изолированных скриншотов (CI): тот же diagnostics-код из core + бутстрап `Application` с FluentTheme + конвенция `[Preview]` (статическая фабрика, отдающая контрол с sample-данными) для рендера компонента в изоляции. 2. **Обогащение `_set` именем свойства.** Генератор уже знает имя свойства — пусть пробрасывает его в `_set` (доп. параметр `property`), чтобы `ViewBuildingException` говорил «while applying 'X'». Это напрямую повышает качество ошибок, которые AgentTools отдаёт агенту, и заодно развязывает диагностику от номера строки.
3. **Sidecar-вариант** как альтернатива в доке: тонкий loopback-эндпоинт в приложении + внешний `dotnet tool` с MCP-сервером, чтобы граф зависимостей приложения оставался чистым. По умолчанию используем in-process пакет; sidecar — для AOT/энтерпрайз-чувствительных потребителей.
